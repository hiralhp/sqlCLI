# Fireworks AI Text-to-SQL CLI

An interactive CLI agent that converts natural language questions into SQLite SQL
and executes them against the Chinook database, built on Fireworks AI via the
OpenAI-compatible Python SDK.

---

## Quick Start

### 1. Prerequisites

- Python 3.11+
- [`uv`](https://docs.astral.sh/uv/) (recommended) — or any venv + pip
- A [Fireworks AI API key](https://fireworks.ai)
- `curl` and `sqlite3` (for the database setup script)

### 2. Set up the database

```bash
cd take-home
bash setup.sh
```

This downloads the Chinook SQLite database to `data/Chinook.db`.

### 3. Configure environment variables

```bash
cp .env.example .env
# Edit .env and set FIREWORKS_API_KEY=<your key>
```

### 4. Install dependencies

```bash
uv sync
```

### 5. Run the CLI

```bash
uv run cli
# or
python -m src.cli
```

### 6. Run the evaluation

```bash
uv run eval
# or
python -m src.eval
```

Results are written to `data/dev_answers.json` (slim format matching the submission
template) and `data/dev_answers_detailed.json` (includes per-question latency).

## Architecture

### `TextToSQLAgent` (`src/agent.py`)

The agent is a thin wrapper around the Fireworks API (accessed via the
OpenAI-compatible SDK).

**Schema injection**
At initialization, the agent calls `get_schema()` and formats every table as a
`CREATE TABLE` statement with column types, `PRIMARY KEY` markers, and
`REFERENCES` annotations derived from `PRAGMA foreign_key_list`. This gives the
model an unambiguous, machine-readable view of every join path.

**Prompt design**
A detailed system prompt is sent on every request containing:
- Hard rules (SELECT only, no invented names, no markdown output)
- The full database schema

Conversation turns (user question → generated SQL) are appended to a rolling
history window (last 5 exchanges = 10 messages) so follow-up questions can
reference prior context.

**Safety guard**
Every generated query is checked: the first token must be `SELECT`. Anything
else is rejected before hitting the database.

**Retry-with-repair loop**
If the database raises an error, the agent sends the broken SQL and the error
message back to the model and asks for a corrected query. This repeats up to
`max_retries` times (default 2) before surfacing the error to the caller.

**Model choice**
`qwen3-235b-a22b-instruct-2507` — a large instruction-tuned MoE model with
strong SQL reasoning, available on Fireworks at a fraction of GPT-4 cost and
with latency well under the 3 s P50 target for typical Chinook queries.

### `CLI` (`src/cli.py`)

- Loads `.env`, connects to the DB, and instantiates the agent.
- REPL loop: reads a line, calls `agent.ask()`, prints the SQL and results as
  an ASCII table.
- History is automatically maintained across turns by the agent; type `exit` or
  `quit` to end the session.

### `Eval` (`src/eval.py`)

- Loads `data/dev_questions.json` (10 questions, tiers 1–3).
- Calls `agent.ask()` for each question, times the round-trip with
  `time.perf_counter()`.
- Calls `agent.reset_history()` between questions so each is evaluated
  independently.
- Writes `data/dev_answers.json` (submission format) and
  `data/dev_answers_detailed.json` (includes `latency_s`).

## Evaluation Notes

The 10 development questions span three complexity tiers:

| Tier | Description | Examples |
|---|---|---|
| 1 | Single-table lookups, simple aggregations | Top genres, albums by artist |
| 2 | Multi-table JOINs, GROUP BY, date filtering | Revenue per year, avg invoice by country |
| 3 | Window functions, subqueries, multi-hop joins | Customer rank, artists in >3 genres |

The agent handles all three tiers reliably with `qwen3-235b-a22b-instruct-2507`
because the full schema (including FK relationships) is present in the prompt,
giving the model everything it needs to construct correct JOIN paths without
hallucinating column names.

To compare against gold-standard answers, see
`data/dev_questions_with_answers.json`.

## Limitations

1. **No streaming** — the CLI waits for the full response before printing.
   Adding `stream=True` to the API call would improve perceived latency for
   longer SQL queries.

2. **Token budget** — `max_tokens=512` is sufficient for the Chinook queries
   tested but could be too small for very complex multi-CTE queries on larger
   schemas.

3. **History is in-memory only** — restarting the CLI clears the conversation.
   Persistent sessions would require saving/loading history to disk.

4. **No result pagination** — large result sets are printed in full. A `LIMIT`
   hint in the prompt or a display cap would improve usability.

5. **Schema is static** — the schema is read once at startup. If the database
   schema changes at runtime the agent needs to be re-initialized.

6. **Single-database** — the CLI is hard-pointed at Chinook. Making the DB path
   a CLI argument (e.g. `uv run cli --db path/to/other.db`) would generalize it.

## Take-Home Context

This project is a response to the Fireworks AI Field Engineering take-home
assessment. The original customer scenario and requirements are reproduced below
for reference.

<details>
<summary>Original take-home brief</summary>

**From:** Raul Jimenez &lt;raul.j@gitlab.com&gt;

GitLab is building an agentic BI CLI for ~1,000 users at ~30 queries/day each.
Current prototype uses a simple prompt with GPT-5.4 and suffers from:

- **Quality** — hallucinated column names, incorrect JOINs, invalid SQL
- **Latency** — ~7 s end-to-end; target is &lt;3 s P50
- **Cost** — GPT-5.4 pricing doesn't scale to 30,000 queries/day

This PoC demonstrates that a schema-aware, retry-capable agent running
`qwen3-235b-a22b-instruct-2507` on Fireworks AI addresses all three pain points.

</details>
